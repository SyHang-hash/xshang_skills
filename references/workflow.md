# 工作流与心智模型

## 完整工作流

```
Phase 1  构思         用户想好要做什么，理清需求
Phase 2  技术选型     AI 给出技术方案（至少3个选项），用户拍板
Phase 3  计划制定     AI 出分模块计划，用户审批确认
Phase 4  分模块执行   AI 按计划逐模块实现，不确定的地方必须问用户
Phase 5  整体测试     所有模块做完，用户做整体测试
Phase 6  查漏补缺     检查漏洞、边界情况、安全问题
Phase 7  文档编写     输出四类文档
Phase 8  提交         代码 + 文档一起提交到 git
```

## 各阶段行为规则

| 阶段 | AI 要做什么 | AI 不能做什么 |
|------|------------|-------------|
| 构思 | 帮用户理清需求边界，给 3 个技术方案 | 直接写代码 |
| 选型 | 输出框架设计（目录结构 + 模块划分 + 技术栈） | 纠结完美，够用就上 |
| 计划 | 把功能拆成独立模块，写清职责和依赖 | 没等确认就开始写 |
| 执行 | 逐模块实现，做完一个汇报一个 | 偷偷加功能、加注释、加"优化" |
| 测试 | 辅助排查，先定位根因再动手 | 盲改、瞎试 |
| 补漏 | 检查安全/边界/代码质量 | 大规模重构 |
| 文档 | 输出四类文档 | 跳过任何一类 |
| 提交 | 代码文档一起提交，规范 commit message | 只提交代码不提交文档 |

## 心智模型

### 1. 人定方向，AI执行，人验收

分工边界清晰：
- **用户的职责**：想清楚做什么（需求定义）、审批计划、验收测试、检查漏洞、最终提交
- **AI的职责**：技术选型、框架构建、制定计划、分模块编码实现
- 协作节奏：用户给方向 → AI出方案 → 用户审批 → AI分模块执行 → 用户整体验收

### 2. 够用就上，迭代优化

不追求一次到位，接受 v1 粗糙但能跑的状态。

典型迭代节奏：
```
第一轮：功能骨架，能跑就行
第二轮：数据对接，确保逻辑正确
第三轮：显示优化，UI 打磨
第四轮：交互增强，体验细节
第五轮：性能优化，安全加固
```

不要想着一步到位。v1 能用就发，有问题下个版本修。
也不要做半截就停。每一轮内的模块要做完整，不留半成品。

### 3. 做一个拉一个

做功能时同步拉通上下游：改了 API 就想想页面要不要同步改，加页面的同时配路由守卫，加功能的同时更新文档。

### 4. 不懂就问，不要猜

AI 在计划和执行过程中遇到不确定的地方，必须主动向用户询问，不得自行猜测或跳过。猜错了返工比多问一句慢得多。

### 5. 修不好就换方案

同一个 bug 修 3 轮还没好，停下来换方案。不要在破方案上继续打补丁。降级、换依赖、换实现方式都行，死磕不行。

## 决策规则清单

| 规则 | 说明 |
|------|------|
| 先跑起来再说 | v1 能用就发，完美方案不如能用的方案 |
| 不懂就问 | 不确定的地方问用户，不要猜 |
| 改一处想三处 | 改代码时考虑上下游影响 |
| 只做计划内的事 | 没说要加的功能就不加 |
| 修3轮换方案 | 同一个问题换思路，不死磕 |
| 文档是交付物 | 功能做完文档也要跟上 |
| 简洁汇报 | 改了什么、影响什么、注意什么，三句话说完 |

## 文档四件套

每次功能完成后，提醒用户更新：

| 类型 | 内容 | 面向谁 |
|------|------|--------|
| **项目总结文档** | 项目全貌、功能清单、亮点展示、统计数据 | 汇报/评审/展示 |
| **技术文档** | 技术架构图（Mermaid）、选型依据、核心实现 | 开发者/技术评审 |
| **模块框架文档** | 各模块职责、接口定义、依赖关系图 | 开发者/维护者 |
| **部署文档** | 环境搭建、Nginx 配置、SSL、上线步骤 | 运维/部署者 |

## 版本管理

```
commit message 格式：
  vX.Y.Z <type>(scope): 描述

type：
  feat / fix / refactor / docs / style / perf / chore / security

版本号：
  +0.0.1  小修补
  +0.1.0  有感知的功能
  +1.0.0  里程碑
```
